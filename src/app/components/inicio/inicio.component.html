Examen Coppel

Necesidad

CRUD Arquitecto de Software

  • Se requiere la generación de pólizas de faltantes, dichas pólizas deben restarse a la cantidad del inventario y ser cargadas a un empleado.
  • Permitir eliminar pólizas que ya no son necesarias.
  • Actualizar los datos del empleado en la póliza en caso de registrarlas con algún error.
  • Generar log para conocer el flujo en caso de algún fallo.

Supuesto

  • Este servicio se debe desplegar de manera centralizada.
  • La base de datos donde se realizarán los grabados para este ejercicio es utilizada por una gran cantidad de procesos que transaccionan, consultan, abren y cierran conexiones, etc.
  • La base de datos es central, toda la operación usa esta única base de datos.

Puntos Técnicos

  • Diseñar solución de Arquitectura:
    • Listado de impacto: Componentes de los cuales se compondrá la solución así como sus características técnicas.
    • Diagrama Arquitectónico: Ambiente donde vivirá cada una de las soluciones (Front, Back y BD). Características de cada uno de los ambientes necesarios para el funcionamiento de la solución (Software, Hardware). Tipo de solución (ejemplo: si es Centralizada, Regional, Central, Clusterizada, etc.).
  • Desarrollo de la solución:
    • FrontEnd: Puede ser en C#, JAVA, Angular. Puedes hacer una APP (opcional). Utilizar la mayor cantidad de controles posibles en pantalla.
    • BackEnd: Desarrollo de servicio web RESTful. Deseable utilizar: SpringBoot + Java 17. Proyecto de Maven. Opcional: RAC (Framework PHP Coppel - Phalcon). Debe considerarse agregar Log. Utilizar JDBCTemplate o similar para la conexión a base de datos y realizar consumo de Procedimientos Almacenados o Funciones dependiendo el motor de base de datos seleccionado. Evitar QUERYS directos hacia la base de datos.
    • Base de Datos: Usar persistencia en BD relacional. Motor de base de dados: SQL Server o PostgreSQL.
  • Almacenamiento de código fuente y versión:
    • Subir a Git personal de Coppel.
  • Documentación Final de Arquitectura:
    • Diagrama de Secuencia.
    • Diagrama de Datos.

Considerar como mínimo los siguientes valores:

  • Creación de Tablas
    • Nombre Tabla: Inventario
      • Campo 1: SKU
      • Campo 2: Nombre
      • Campo 3: Cantidad
    • Nombre Tabla: Empleado
      • Campo 1: IdEmpleado
      • Campo 2: Nombre
      • Campo 3: Apellido
      • Campo 4: Puesto
    • Nombre Tabla: Polizas
      • Campo 1: IdPoliza
      • Campo 2: EmpleadoGenero
      • Campo 3: SKU
      • Campo 4: Cantidad
      • Campo 5: Fecha
  • Notas:
    • No puede existir 2 registros con el mismo SKU
    • No puede existir 2 empleados con el mismo ID
    • No puede existir 2 Pólizas con el mismo ID
